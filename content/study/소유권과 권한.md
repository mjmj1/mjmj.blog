### 기본 개념
기본적으로, **Netcode for GameObjects**는 **클라이언트-서버 토폴로지**(client-server topology)를 전제로 합니다. 이 구조에서는 서버가 모든 `NetworkObject`의 소유자(owners)이며, 생성(spawning)과 제거(despawning)에 대한 최종 권한(authority)을 가집니다.
하지만 Netcode for GameObjects는 또한 **분산 권한 토폴로지**(distributed authority topology)를 지원하며, 이 경우 `NetworkObject`의 소유권과 권한을 더 유연하게 관리할 수 있습니다.
### 소유권(Ownership)과 분산 권한
- 분산 권한 환경(distributed authority setting)에서는 `NetworkObject`에 대한 권한(authority)이 단일 서버에만 묶여 있는 것이 아니라, **해당 객체의 소유권 권한 설정(ownership permission settings)에** 따라 클라이언트들에게 분산됩니다.
- 소유권이 분산 가능(distributable)한 권한 설정이 되어 있는 객체들은 클라이언트가 접속(join)하거나 접속 해제(disconnect)할 때 자동으로 분산되어 할당됩니다.
- 세션이 분산 권한 모드로 시작되면:
    1. 각 클라이언트는 자신의 플레이어 객체(player)를 생성(spawn)합니다.
    2. 로컬 플레이어(local player)의 소유권(permission)을 잠금(lock)하여 다른 클라이언트가 소유권을 가져가지 못하도록 합니다.
    3. 이후엔 일부 `NetworkObject`들을 생성(spawn)합니다.  
        이 시점에서는 클라이언트 A가 distributable 설정된 객체들 및 그 클라이언트 자신의 플레이어 객체에 대한 완전한 권한(full authority)을 가집니다.
- 새로운 플레이어가 접속하면(distributed authority network topology하에서), 분산 가능(distributable)한 객체들의 권한이 기존 클라이언트들과 나눠집니다. 이렇게 함으로써 각 클라이언트의 처리 부담(processing)과 대역폭(bandwidth) 부담을 줄일 수 있습니다.
- 플레이어가 떠날 경우(gracefully 또는 연결 끊김), 그 플레이어가 소유하던 객체의 소유권 및 마지막 알려진 상태(last known state)는 게임플레이의 중단 없이(“without interruption”) 나머지 연결된 클라이언트들에게 이전됩니다.
### 소유권 허가 설정 (Ownership permission settings) — 분산 권한에서만 해당됨
`NetworkObject.OwnershipStatus`라는 설정을 통해 분산 권한 네트워크 토폴로지 사용 시 객체의 소유권이 어떻게 관리되는지를 정할 수 있습니다. 주요 옵션은 다음과 같습니다.

| 설정 이름               | 설명                                                                                                                                                                                                                                                           |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **None**            | 이 `NetworkObject`의 소유권이 정적(static)으로 간주되어, 재분배(distribution), 요청(request), 또는 이전(transfer)이 불가능함. 예: 플레이어 객체(player object) 등이 이에 해당할 수 있음. [Unity Docs](https://docs.unity3d.com/Packages/com.unity.netcode.gameobjects%402.5/manual/basics/ownership.html) |
| **Distributable**   | 클라이언트가 접속하거나 나갈 때 소유권이 자동으로 재분배됨, 단 소유권이 잠겨 있거나(locked) 이전 요청(request pending)이 없는 조건 하에서. [Unity Docs](https://docs.unity3d.com/Packages/com.unity.netcode.gameobjects%402.5/manual/basics/ownership.html)                                                  |
| **Transferable**    | 소유권 이전(transfer)이 가능함, 단 소유권이 잠겨 있지 않고 이전 요청이 대기 중이지 않은 경우에만 즉시 이전할 수 있음. [Unity Docs](https://docs.unity3d.com/Packages/com.unity.netcode.gameobjects%402.5/manual/basics/ownership.html)                                                                   |
| **RequestRequired** | 소유권 이전 전 반드시 요청을 해야 하며, 이전이 완료된 후엔 항상 소유권이 잠김(lock) 상태가 됨. [Unity Docs](https://docs.unity3d.com/Packages/com.unity.netcode.gameobjects%402.5/manual/basics/ownership.html)                                                                                  |
| **SessionOwner**    | 이 `NetworkObject`는 **세션 소유자(session owner)**가 항상 소유함. 이전되거나 분산되지 않음. 단, 세션 소유자가 변경되면 이 객체는 자동으로 새 세션 소유자에게 이전됨. [Unity Docs](https://docs.unity3d.com/Packages/com.unity.netcode.gameobjects%402.5/manual/basics/ownership.html)                             |

- `NetworkObject.SetOwnershipLock`을 사용하면 임시적으로 객체의 소유권 변경을 막기 위해(permission settings을 잠그거나 잠금 해제) 할 수 있음.
- **주의(Note)**:
    - 이 소유권 허가(permission) 설정들은 에디터(Editor) 내에서 `NetworkObject`를 검사(inspect)할 때만 보임. Multiplayer Services SDK 패키지가 설치된 경우에만.
    - 클라이언트-서버 토폴로지를 사용 중일 경우엔 이러한 소유권 설정들이 실질적 영향을 주지 않음. 왜냐하면 서버가 항상 권한(authority)을 갖고 있으므로.        
    - 분산 권한 토폴로지를 사용해야만 소유권(permission)이 유효함.

### 권한 검사 (Checking for authority)
- `HasAuthority` 속성(property)은 `NetworkObject`와 `NetworkBehaviour` 둘 다에서 사용할 수 있으며, 분산 권한 분포(distributed authority) 모드이든, 클라이언트-서버(client-server) 모드이든 상관없이 작동함. 개별 객체(object)에 대해 작업할 때는 권한이 있는지를 확인할 때 이걸 사용하는 것이 권장됨.
    예시 코드가 문서에 있음.
- 분산 권한 모드에서는 권한(authority)이 “현재 권한을 가진 클라이언트(client instance)”에 속함. 따라서 객체의 상태(state)를 저장하거나 소유권이 다른 클라이언트로 이전되는 경우, 또는 모든 클라이언트가 세션에서 나갈 경우에도 상태가 유지되도록 하기 위해 **NetworkVariables**를 사용하는 것이 필요함. 지역(local), 비복제(non-replicated) 속성(properties)에만 의존해서는 안 됨.
- NetworkVariables는 내장된(built-in) 모든 데이터 타입을 지원하며, 분산 권한 세션(distributed authority session)에서는 “소유자 쓰기(owner write)” 및 “모두 읽기(everyone read)” 권한 설정이 자동으로 적용됨. 서버가 권위(authoritative server) 라는 개념이 없기 때문임.
---
### `IsServer`와의 비교
- `IsServer` 또는 `!IsServer` 체크는 전통적인 클라이언트-서버 구조에서 현재 부분이 서버인지 여부를 판단하기 위해 사용됨. 하지만 분산 권한 게임에서는 “하나의 권위 있는 서버” 개념이 없기 때문에 `IsServer`가 항상 의미 있는 것은 아님.
- 클라이언트-서버 게임을 만드는 경우엔 `HasAuthority`와 `IsServer`를 혼합해서 사용하는 것이 가능하고, 객체 수준(object-specific) 작동에는 `HasAuthority`가, 전역 작업(global actions)에는 `IsServer`가 유용할 수 있음. 예시도 문서에 있음.

## 요약
- 기본적으로 Netcode for GameObjects는 서버가 모든 객체를 소유하고 관리하는 구조를 가짐.
- 하지만 **분산 권한**(distributed authority)을 사용하면, 소유권/권한이 클라이언트 간에 나뉘며 `NetworkObject`마다 설정된 소유권 허가(permission status)에 따라 어떻게 분배되거나 이전될 수 있는지 결정됨.
- `HasAuthority`를 통해 어떤 클라이언트가 해당 객체에 현재 권한이 있는지를 코드에서 항상 체크해야 함.
- 객체의 상태(state)의 동기화 및 일관성을 위해 NetworkVariables를 사용하는 것이 중요함.
- 소유권(permission) 및 잠금(lock) 설정을 통해 객체의 소유권 변화(transfer, request, 분산, etc.)를 제어 가능.
